# 의견청취 처리 플로우 — 개발 스펙

> 작성일: 2026-04-27
> 관련 메뉴: 의견청취 / 안전점검 / 개선조치
> 우선순위: 신규 개발

---

## 1. 개요

### 변경 배경

기존 의견청취는 접수 → 검토중 → 조치중 → 완료의 단순 상태 변경 구조로 개발되어 있으나, 아래 문제가 있어 재설계한다.

- 검토중 단계가 실질적으로 아무 행위도 없는 빈 상태
- 상태 변경이 단순 버튼 클릭이라 조치 내용이 기록되지 않음
- 현장점검 / 개선조치가 별도 메뉴와 연동되지 않음

### 핵심 변경사항

| 항목          | 기존           | 변경                                  |
| ------------- | -------------- | ------------------------------------- |
| 검토중 상태   | 존재           | **제거**                              |
| 접수 → 조치   | 단순 상태 변경 | **조치 방법 선택 + 하위 데이터 입력** |
| 현장점검 연동 | 없음           | **안전점검 특별점검으로 자동 생성**   |
| 개선조치 연동 | 없음           | **개선조치 자동 생성 + 출처 연결**    |

## 2. 상태 정의

### 상태 목록

| 상태         | 설명                              | 다음 상태                  |
| ------------ | --------------------------------- | -------------------------- |
| `접수대기`   | 근로자가 등록한 초기 상태         | 점검중 / 개선조치중 / 반려 |
| `점검중`     | 현장점검 실시 선택 후 저장된 상태 | 개선조치중 / 완료          |
| `개선조치중` | 개선조치 생성 후 조치 진행 중     | 완료                       |
| `완료`       | 최종 처리 완료 (자동 전환)        | —                          |
| `반려`       | 반려 처리 (수동)                  | —                          |

### 상태 전이 다이어그램

```
접수대기
  ├── [현장점검 선택 + 저장] ──→ 점검중
  │     ├── [점검 완료 + 개선조치 필요] ──→ 개선조치중
  │     │     └── [개선조치 완료] ──→ 완료 (자동)
  │     └── [점검 완료 + 이상 없음] ──→ 완료 (자동)
  │
  ├── [개선조치 선택 + 저장] ──→ 개선조치중
  │     └── [개선조치 완료] ──→ 완료 (자동)
  │
  └── [반려 선택 + 저장] ──→ 반려
```

## 3. 처리 플로우 상세

### 3-1. 접수대기 화면

**접근 권한**: 안전관리자, 사업장관리자

**화면 구성**:

- 의견청취 기본 정보 (분류, 제목, 작성자, 작성일, 사업장, 상세내용, 첨부사진)
- 처리하기 버튼 → 조치 선택 화면으로 이동

**작업중지요청 한정 추가 동작**:

- 접수 즉시 담당 안전관리자에게 알림톡 자동 발송 (긴급 표시)

---

### 3-2. 조치 선택 화면

**접수대기 상태에서만 진입 가능**

**선택 옵션 3가지 (라디오 선택)**:

#### A. 현장점검 실시

선택 시 입력 폼 노출:

| 필드             | 타입        | 필수 | 비고                                           |
| ---------------- | ----------- | ---- | ---------------------------------------------- |
| 점검 종류        | select      | ✅   | 일상/주간/합동/순회/특별 — **특별점검 기본값** |
| 점검 예정일      | date        | ✅   |                                                |
| 담당자           | user search | ✅   | 조직도 내 검색                                 |
| 점검 시 확인사항 | textarea    | ❌   | 의견청취 내용 자동 복사 (수정 가능)            |

저장 시 동작:

1. 의견청취 상태 → `점검중`
2. **안전점검 메뉴에 특별점검 레코드 자동 생성** (아래 3-3 참조)
3. 처리 이력 기록

#### B. 개선조치 바로 생성

선택 시 입력 폼 노출:

| 필드        | 타입        | 필수 | 비고                                |
| ----------- | ----------- | ---- | ----------------------------------- |
| 담당자      | user search | ✅   | 조직도 내 검색                      |
| 완료 예정일 | date        | ✅   |                                     |
| 개선 내용   | textarea    | ✅   | 의견청취 내용 자동 복사 (수정 가능) |

저장 시 동작:

1. 의견청취 상태 → `개선조치중`
2. **개선조치 메뉴에 레코드 자동 생성** (아래 3-4 참조)
3. 처리 이력 기록

#### C. 반려

선택 시 입력 폼 노출:

| 필드      | 타입     | 필수 | 비고                   |
| --------- | -------- | ---- | ---------------------- |
| 반려 사유 | textarea | ✅   | 작성자 알림톡에 포함됨 |

저장 시 동작:

1. 의견청취 상태 → `반려`
2. 작성자에게 반려 사유 포함 알림톡 자동 발송
3. 처리 이력 기록

---

### 3-3. 안전점검 자동 생성 스펙 (현장점검 선택 시)

**트리거**: 의견청취 처리에서 "현장점검 실시" 선택 후 저장

**생성 위치**: 안전점검 메뉴 > 점검 목록

**자동 생성 데이터**:

| 안전점검 필드    | 값                             |
| ---------------- | ------------------------------ |
| 점검명           | `[의견청취] {의견청취 제목}`   |
| 점검 종류        | 특별점검 (기본값, 변경 가능)   |
| 점검 예정일      | 의견청취 처리 시 입력한 예정일 |
| 점검 담당자      | 의견청취 처리 시 선택한 담당자 |
| 점검 장소        | 사업장 (의견청취 소속 사업장)  |
| 출처             | `의견청취 연동` 배지 표시      |
| 출처 링크        | 해당 의견청취 레코드 ID 참조   |
| 점검 시 확인사항 | 의견청취 처리 시 입력한 내용   |

**연동 관계**:

- 안전점검 레코드에 `의견청취_id` FK 저장
- 의견청취 상세에 연결된 안전점검 카드 표시 (점검명 / 상태 / 예정일)

**점검 완료 후 분기**:

```
점검 완료 버튼 클릭
├── 개선조치 필요 선택
│   → 개선조치 정보 입력 (담당자 / 완료예정일 / 개선내용)
│   → 저장 시: 의견청취 상태 → 개선조치중
│             개선조치 메뉴에 레코드 자동 생성 (출처: 안전점검 연동)
│
└── 이상 없음 선택
    → 저장 시: 의견청취 상태 → 완료 (자동)
              처리 이력 기록
```

---

### 3-4. 개선조치 자동 생성 스펙

**트리거 2가지**:

1. 의견청취에서 "개선조치 바로 생성" 선택 후 저장
2. 의견청취 → 현장점검 완료 후 "개선조치 필요" 선택 후 저장

**생성 위치**: 개선조치 메뉴 (GNB 독립 메뉴)

**자동 생성 데이터**:

| 개선조치 필드 | 케이스 1 (직접 생성) | 케이스 2 (점검 후 생성) |
| ------------- | -------------------- | ----------------------- |
| 출처 배지     | `의견청취`           | `안전점검`              |
| 출처 링크     | 의견청취 레코드 ID   | 안전점검 레코드 ID      |
| 담당자        | 입력한 담당자        | 입력한 담당자           |
| 완료 예정일   | 입력한 날짜          | 입력한 날짜             |
| 개선 내용     | 입력한 내용          | 입력한 내용             |
| 상태          | 조치중               | 조치중                  |

**개선조치 완료 시 자동 동작**:

- 개선조치 상태 → `완료`
- **연결된 의견청취 상태 → `완료` 자동 전환**
- 처리 이력에 "개선조치 완료 → 자동 완료 처리" 기록

## 4. 처리 이력 스펙

모든 상태 변경 시 이력 기록 필수.

| 필드      | 내용                     |
| --------- | ------------------------ |
| 상태      | 변경된 상태명            |
| 처리 내용 | 선택한 조치 방법 및 요약 |
| 처리자    | 로그인한 사용자          |
| 처리 일시 | 서버 타임스탬프          |

**이력 예시 (현장점검 → 개선조치 경로)**:

```
1. 접수          2026-04-21 14:32  하늘
2. 현장점검 생성  2026-04-21 15:10  김관리  (점검중)
3. 점검 완료     2026-04-22 11:00  김관리  → 개선조치 필요
4. 개선조치 생성  2026-04-22 11:00  시스템  (개선조치중)
5. 개선조치 완료  2026-04-30 09:00  박시설
6. 자동 완료     2026-04-30 09:00  시스템  (완료)
```

## 7. DB 설계 참고

### 의견청취 (opinion_requests)

```sql
id                  UUID PK
site_id             UUID FK (사업장)
category            ENUM('작업중지요청','청취조사','아차사고','안전제안','위험신고')
title               VARCHAR(50)
content             TEXT
attachments         JSON (파일 URL 배열)
status              ENUM('접수대기','점검중','개선조치중','완료','반려')
created_by          UUID FK (users)
created_at          TIMESTAMP

-- 처리 정보
processed_by        UUID FK (users) NULLABLE
reject_reason       TEXT NULLABLE

-- 연동 FK
linked_inspection_id    UUID FK (safety_inspections) NULLABLE
linked_improvement_id   UUID FK (improvements) NULLABLE
```

### 처리 이력 (opinion_request_histories)

```sql
id                  UUID PK
opinion_request_id  UUID FK
status              ENUM (위와 동일)
action_description  VARCHAR(200)
processed_by        UUID FK (users)
created_at          TIMESTAMP
```

### 안전점검 (safety_inspections) — 기존 테이블 컬럼 추가

```sql
-- 기존 컬럼 유지 +
source_type         ENUM('직접등록','의견청취연동') DEFAULT '직접등록'
source_opinion_id   UUID FK (opinion_requests) NULLABLE
```

### 개선조치 (improvements) — 기존 테이블 컬럼 확인

```sql
source_type         ENUM('위험성평가','안전점검','의견청취')
source_id           UUID (각 출처 테이블의 ID)
```

## 8. 구현 시 주의사항

1. **개선조치 완료 → 의견청취 자동 완료**: 개선조치 상태가 `완료`로 변경될 때, `source_type`이 `의견청취` 또는 `안전점검`(의견청취 연동)인 경우 연결된 의견청취 상태를 `완료`로 자동 전환하는 트리거 필요

2. **안전점검 목록에서 출처 표시**: `source_type`이 `의견청취연동`인 경우 목록 컬럼에 배지 표시 (필터 옵션에도 추가)

3. **점검 완료 UI**: 안전점검 메뉴에서 점검 완료 처리 시, 해당 점검이 의견청취 연동인 경우 "개선조치 필요 여부" 선택 UI 추가 노출

4. **의견청취 상세에서 연동 카드 표시**:
   - 점검중 상태: 연결된 안전점검 카드 (점검명 / 상태 / 예정일 / 바로가기)
   - 개선조치중 상태: 연결된 개선조치 카드 (내용 / 담당자 / 완료예정일 / 상태 / 바로가기)

5. **반려 후 재처리**: 반려된 의견청취는 재처리 불가 (읽기전용). 필요 시 작성자가 신규 등록
